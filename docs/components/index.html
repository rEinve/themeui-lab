Components

Core components are reusable UI pieces that style their own interior and state only.

Canonical component classes are .button, .card, .field-label, .field-hint, .field-error, .field-success, .field-warning, .input, .select, .textarea, .radio, .checkbox, .switch, .tag, .divider, .list, .pagination, .breadcrumbs, .tabs, .carousel, and .table. The legacy .btn alias remains temporarily for compatibility only.

Table

The first pass is a styled native table. Keep data structure semantic and use a wrapper only for horizontal overflow.

Contract: use real table elements in order: table, optional caption, thead, tbody, tr, and scoped header cells. Keep .table-wrapper only as the overflow shell, not as a replacement for semantic table structure.

Do

Keep headers scoped and use the wrapper only when the table needs horizontal scrolling.

Don't

Use div rows and columns to imitate data tables that already have a native semantic element.

Content List

Use the list component for repeated content blocks that need consistent rhythm, titles, and supporting text.

Contract: .list is only the rhythm shell. Choose the inner element by meaning: use li inside a semantic list, article for standalone repeated entries, or div only when there is no stronger HTML element. Keep .list__title and .list__text optional.

Do

Pick the repeated item element by meaning, then add .list__title and .list__text only when needed.

Don't

Treat .list as a page layout primitive or assume every item must be an article.

Design tokens

Foundational decisions stay centralized and predictable for downstream teams.

Composable primitives

Layouts remain flexible because repeated content is handled by the list, not by page shell abstractions.

Reusable components

Cards, buttons, and form controls can sit inside or alongside list content without special variants.

<div class="list">
  <article class="list__item">
    <h3 class="list__title">Design tokens</h3>
    <p class="list__text">Foundational decisions stay centralized and predictable for downstream teams.</p>
  </article>
  <article class="list__item">
    <h3 class="list__title">Composable primitives</h3>
    <p class="list__text">Layouts remain flexible because repeated content is handled by the list, not by page shell abstractions.</p>
  </article>
</div>

FAQ Composition

FAQ can stay as composition. Use list for repeated entries and keep the surrounding page structure in primitives.

Frequently asked questions

A simple product FAQ built from a section header plus the list component.

How long does setup take?

Most teams can get started in less than a day with the current foundations and component set.

Can we use our own tokens?

Yes. The system is designed around token-driven foundations and exported CSS custom properties.

Do we need a special FAQ component?

Not yet. Repeated editorial content can stay on the neutral list component until a clearer need appears.

Breadcrumbs

Breadcrumbs stay text-first and low-emphasis: a simple trail with links for prior levels and aria-current for the current page.

Contract: wrap breadcrumbs in a labeled nav, keep the trail in an ordered list, and use aria-current="page" on the final item only. Prior levels should stay real links when navigation is possible.

Do

Keep only the current page non-linked and mark that final crumb with aria-current="page".

Don't

Turn every crumb into plain text or mark multiple items as current.

Pagination

Pagination stays small: a nav region, a list, link elements, and an aria-current state for the active page.

Contract: wrap pagination in a labeled nav and keep items in a list. Use aria-current="page" on the active item. If a control cannot be activated, prefer a button with disabled over a placeholder link.

Do

Use links for real destinations and a disabled button when previous or next is temporarily unavailable.

Don't

Ship fake disabled links that still look or announce like navigation targets.

Tabs

Tabs work well for a few peer views. Styling stays in the component, while a tiny vanilla JS helper manages keyboard navigation and panel visibility.

Contract: keep data-tabs on the root, one [role="tablist"], buttons with role="tab", and panels with role="tabpanel". Each tab needs a unique id and matching aria-controls; each panel needs a matching id and aria-labelledby.

Do

Use tabs for a small set of peer views that share the same surrounding context.

Don't

Use tabs as a replacement for page layout, long multi-step flows, or missing heading structure.

Workspace overview

Keep the first panel visible in markup so the content still reads naturally before JavaScript enhances the experience.

Recent activity

Arrow keys move between tabs, and the active panel is the only one shown after initialization.

Workspace settings

Use tabs for short sibling views, not for broad page composition or multi-step flows.

Carousel

Use a carousel for a short sequence of peer slides. The component owns its viewport, controls, and state, while content inside each slide stays flexible.

Contract: keep data-carousel on the root, data-carousel-track on the moving track, and data-carousel-slide on each slide. Previous/next buttons and dot buttons are optional, but if present they must use the matching data-carousel-prev, data-carousel-next, and data-carousel-dot hooks. Hidden slides are made non-tabbable by the initializer, so slide content can include links or buttons safely.

Do

Keep carousels short, give the region a clear label, and make each slide meaningful on its own.

Don't

Bury essential content in long sliders or assume motion alone will communicate state.

Release note

Small, composable foundations

Ship layout primitives, then layer reusable components on top instead of baking page structure into every feature.

Accessibility

Keyboard-friendly controls

Previous, next, and direct slide buttons keep the interaction predictable without adding a heavy dependency.

Implementation

Vanilla JS only where needed

A tiny initializer updates the active slide, disabled states, and status text while the HTML remains readable on its own.

Button + Tag

Components should stay small and reusable, then compose together inside flow primitives.

Contract: use .tag for short status or metadata labels only. Tags are presentational text by default, not interactive controls, and they work best when the content is brief enough to avoid wrapping or truncation pressure in tight layouts.

Do

Use tags for short labels like status, category, or environment markers.

Don't

Use a tag as a button, filter chip, or home for long free-form text.

Form Control Family

A practical form needs consistent labels, hints, grouped choices, and a clear validation contract built on semantic status tokens.

Contract: every input should have a programmatic label. Prefer explicit label for plus control id, then connect help or validation text with aria-describedby. For invalid fields, pair aria-invalid="true" with an error element and, when possible, aria-errormessage.

Do

Connect labels, hints, and errors programmatically so the visual grouping and accessibility tree match.

Don't

Rely on placeholder text as the label or show validation copy without associating it to the control.
