Components

Core components are reusable UI pieces that style their own interior and state only.

Canonical component classes are .button, .card, .field-label, .field-hint, .field-error, .field-success, .field-warning, .input, .select, .textarea, .radio, .checkbox, .switch, .tag, .divider, .list, .pagination, and .breadcrumbs. The legacy .btn alias remains temporarily for compatibility only.

Content List

Use the list component for repeated content blocks that need consistent rhythm, titles, and supporting text.

Design tokens

Foundational decisions stay centralized and predictable for downstream teams.

Composable primitives

Layouts remain flexible because repeated content is handled by the list, not by page shell abstractions.

Reusable components

Cards, buttons, and form controls can sit inside or alongside list content without special variants.

FAQ Composition

FAQ can stay as composition. Use list for repeated entries and keep the surrounding page structure in primitives.

Frequently asked questions

A simple product FAQ built from a section header plus the list component.

How long does setup take?

Most teams can get started in less than a day with the current foundations and component set.

Can we use our own tokens?

Yes. The system is designed around token-driven foundations and exported CSS custom properties.

Do we need a special FAQ component?

Not yet. Repeated editorial content can stay on the neutral list component until a clearer need appears.

<section class="stack stack--sm">
  <div class="stack stack--xs">
    <h2>Frequently asked questions</h2>
    <p>A simple product FAQ built from a section header plus the list component.</p>
  </div>
  <div class="list">
    <article class="list__item">
      <h3 class="list__title">How long does setup take?</h3>
      <p class="list__text">Most teams can get started in less than a day.</p>
    </article>
  </div>
</section>
